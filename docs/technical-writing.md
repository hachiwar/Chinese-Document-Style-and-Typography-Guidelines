# 技术文档专项规范

本规范适用于项目 README、开发者文档、API 文档、操作手册和故障排查文档。

## 1. 任务导向

技术文档应围绕读者需要完成的任务组织内容，而不是按照系统内部模块简单罗列功能。

操作型文档推荐采用以下结构：

1. 目标和适用场景；
2. 前置条件；
3. 操作步骤；
4. 预期结果；
5. 验证方法；
6. 常见错误与恢复方式；
7. 后续步骤。

## 2. README

项目 README 推荐包含：

- 项目解决的问题；
- 最小可运行示例；
- 安装与使用方法；
- 环境要求；
- 配置说明；
- 测试方法；
- 贡献方式；
- 许可证。

安装和使用入口应位于较前位置。项目历史、设计背景和扩展说明不应阻碍读者查找基本使用方法。

## 3. 操作步骤

- 每个步骤只执行一个主要动作；
- 步骤以动词开头；
- 动作前说明必要的前置条件；
- 动作后说明可观察的成功结果；
- 分支条件紧跟相关步骤；
- 超过 9 步时拆分为多个阶段。

示例：

1. 运行以下命令安装依赖：

   ```powershell
   npm install
   ```

2. 确认命令退出码为 `0`。
3. 运行 `npm test` 验证安装结果。

## 4. 命令与代码

### 4.1 可复制性

- 不在命令前加入 `$`、`>` 等提示符，除非明确说明不需要复制；
- 将命令与输出放入不同代码块；
- 标注适用的 Shell 或编程语言；
- 多平台命令分别提供；
- 不将 Windows 和 Unix 命令混入同一代码块。

### 4.2 占位符

占位符应具有明确形式：

```bash
curl -H "Authorization: Bearer YOUR_API_KEY" \
  https://api.example.com/v1/users
```

代码块后应说明：

> 将 `YOUR_API_KEY` 替换为实际 API 密钥，不保留占位符本身。

### 4.3 示例完整度

示例必须说明其性质：

- 完整可运行示例；
- 仅展示关键逻辑；
- 伪代码；
- 省略错误处理；
- 仅适用于测试环境。

不得使读者自行判断示例能否直接用于生产环境。

## 5. API 文档

每个接口至少应说明：

- 请求方法与路径；
- 接口功能；
- 鉴权要求；
- 路径、查询、请求头和请求体参数；
- 参数类型、是否必填、默认值和限制；
- 请求示例；
- 成功响应；
- 错误响应与错误码；
- 幂等性、分页、限流和版本策略；
- 安全注意事项。

参数表中的同类列名必须统一，例如始终使用“参数、类型、必填、默认值、说明”。

## 6. 故障排查

故障条目推荐采用固定结构：

1. **现象**：用户观察到的结果；
2. **原因**：问题产生的原因；
3. **诊断**：确认原因的方法；
4. **解决**：修复步骤；
5. **验证**：确认恢复的方法；
6. **预防**：避免再次发生的措施。

错误信息应按界面原文引用：

> 错误信息：`ECONNREFUSED 127.0.0.1:5432`

只写“连接失败”会降低可检索性，也不利于读者判断是否属于同一问题。

## 7. 风险操作

涉及删除、覆盖、迁移、权限、付费和生产环境的操作，必须：

- 在操作前说明具体后果；
- 明确是否可逆；
- 提供备份或快照方法；
- 提供回滚步骤；
- 标明所需权限；
- 避免使用“可能有影响”等模糊表述。

不规范写法：

> 注意：执行此命令可能有风险。

规范写法：

> 警告：此命令会永久删除 `users` 表及其全部数据，且无法通过应用程序恢复。执行前必须创建数据库快照。

## 8. 版本与时效性

- 写明文档适用版本；
- 对已弃用功能标明弃用版本、替代方案和计划移除版本；
- 不使用缺少日期的“目前”“最新”“近期”等相对表述；
- 必须使用相对时间时，同时写明绝对日期；
- 发布后定期检查版本、链接、截图和命令。

## 9. 可用性测试

操作型文档发布前，推荐由一位不熟悉该功能的测试者按照文档完整执行，并记录：

- 需要猜测的步骤；
- 未定义的术语；
- 不明确的成功信号；
- 缺少解决方法的错误；
- 出现位置过晚的前置条件。

如果测试者必须向作者提问才能完成任务，文档仍需修订。
